## Objective
Define the exact integration protocol between the AI Team's bot and the AWS/Scraping Team's data pipeline. This is the single source of truth for how data flows from the DPSA website to the bot's knowledge base. Both teams must agree on every interface defined here.

---

## Overview
The AWS/Scraping Team owns the entire data supply chain. The AI Team consumes their output. There is no polling — the system is event-driven. When the AWS scraper finishes a run, it fires an S3 Event Notification. The bot responds immediately, re-querying Postgres/pgvector for fresh data.

---

## 1. Data Flow Diagram
```
DPSA Website
    → AWS EventBridge Cron
        → ECS Scraper (AWS Team)
            → S3 Raw Storage (PDF + HTML snapshots)
                → AWS Text Extraction + Chunking + Embedding
                    → Postgres / pgvector index (AWS Team owns)
                        → S3 Event Notification FIRES
                            → bot/tools/reindexer.py (AI Team)
                                → Bot knowledge base updated instantly
```

---

## 2. Agreed Data Schema — Postgres/pgvector

**Table:** `document_chunks`

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid | Unique chunk identifier |
| `content` | text | The actual text chunk |
| `embedding` | vector(1536) | pgvector embedding for semantic search |
| `source_url` | text | Original DPSA page or PDF URL |
| `source_title` | text | Document or page title |
| `category` | text | `employment`, `benefits`, `policies`, `procedures` |
| `doc_type` | text | `pdf`, `html`, `csv` |
| `scraped_at` | timestamp | When this chunk was last scraped/updated |
| `language` | text | Source language (default: `en`) |

**Status:** DRAFT — must be reviewed and signed off by both teams before implementation.

## 3. S3 Event Notification Specification

### Trigger condition
Fires automatically when the AWS scrape pipeline completes and new files land in S3.

### Expected payload format (to be confirmed with AWS team)
```json
{
  "source": "aws.s3",
  "bucket": "dpsa-knowledge-base",
  "key_prefix": "raw/",
  "scrape_run_id": "run_20260306_020000",
  "scrape_timestamp": "2026-03-06T02:00:00Z",
  "files_added": 14,
  "files_updated": 3
}
```

### Bot's listener
- `tools/reindexer.py` must be running and listening for this trigger event
- On receipt, immediately execute `document_ingest.md` workflow
- Respond with HTTP 200 to acknowledge receipt before starting the re-index

## 4. pgvector Query Interface

The AI Team's `tools/retriever.py` uses the following query pattern. The AWS Team must ensure their pgvector schema supports this:

```sql
SELECT content, source_url, source_title, category, doc_type, scraped_at
FROM document_chunks
ORDER BY embedding <=> $1  -- cosine similarity
LIMIT $2;
```

Where:
- `$1` = query embedding vector (1536 dimensions, generated by AI team's embedder)
- `$2` = `MAX_RETRIEVAL_CHUNKS` from `.env`

**Embedding model:** Must match between the AI Team's query embedder and the AWS Team's ingestion embedder. Confirm embedding model name before go-live.

## 6. Error Escalation Protocol

If the AI Team encounters issues with the data supplied by the AWS Team:

1. Document the issue in the **Known Issues** section below with: date, symptom, affected chunks/files
2. Do NOT modify or patch data supplied by the AWS Team — only document the issue
3. Raise the issue with the AWS Team through the agreed communication channel (TBD)
4. AWS Team resolves on their side and fires a fresh S3 trigger
5. AI Team validates fix on next re-index

---

## 7. Change Protocol

Any changes to the data schema (Section 2) or trigger payload format (Section 3) must be:
1. Proposed in writing by the initiating team
2. Reviewed and approved by both teams
3. Updated in this document before implementation
4. Tested in a staging environment before production
